The current state
The answer to the first two is yes. The main upstream OCaml implementation is open source, published by the OCaml project, with source code in the public ocaml/ocaml repository. The official site also lists current public releases; as of now it shows OCaml 5.4.1 as the latest release and 4.14.3 as the LTS release. 

The upstream implementation is not a toy interpreter. It includes:

a bytecode compiler

a bytecode runtime/interpreter written in C

a native-code compiler

the runtime system and standard libraries 

So the main implementation is already open source, and there are also public forks and variants built from it. The important point is that you do not need to hunt for some separate “open” implementation: the primary one is already open. 

Is it self-hosting?
Mostly yes, but not in the pure Lisp/Forth sense.

The OCaml compiler is bootstrapped. The project’s own hacking docs say the compiler is bootstrapped and that previously compiled bytecode versions of the compiler and lexer are kept in the repository under boot/. Those are used after the C bytecode runtime is built, and then a fresh compiler is built from there. 

That means:

the compiler is substantially written in OCaml

but it still depends on a runtime written in C

and the initial build uses prebuilt bootstrap artifacts rather than “compile from handwritten source using only assembly” 

So compared with your examples:

More self-hosting than many C-family compilers in practice

Less minimal/purely circular than classic small Lisp or Forth systems that can often be grown from a tiny seed interactively

What language should you implement it in?
Best practical answer: C, Rust, or OCaml
C
This is the most traditional choice, and it matches part of the upstream runtime model.

Why choose C

simple portability story

easy to interface with low-level runtime and GC work

classic compiler literature maps well to it

easy to bootstrap from almost anywhere

Downside

you spend more effort on memory safety and tooling discipline

Rust
This is probably the best modern choice for a fresh implementation if you are comfortable in it.

Why choose Rust

safer runtime/compiler implementation

Downside

GC/runtime implementation can feel more awkward than in C

Does it bootstrap like Lisp or Forth?
Similarity
Yes, in the sense that later generations of the compiler are built by earlier generations of the compiler. That is the classic bootstrap/self-hosting idea. OCaml explicitly documents that its compiler is bootstrapped. 

Difference
No, in the sense that it is not usually grown from a tiny interactive nucleus the way many Lisp or Forth systems historically were.

OCaml’s bootstrap path depends on:

prebuilt bootstrap compiler artifacts in boot/

a C runtime

a more conventional build chain 

So the best description is:

OCaml is self-hosting/bootstrapped, but not “minimal-seed interactive bootstrap” in the classic Lisp/Forth style.

Practical recommendation
If you want to create one from scratch, I would not start with “full OCaml.”

I would build this sequence:

mini-ML

integers, booleans, functions, let, tuples

add HM type inference

add variants + pattern matching

add closures and a small runtime

add modules later, or skip them entirely for a long time

decide whether you want:

interpreter

bytecode VM

native backend

That gives you something educational and tractable.

Bottom line
Yes, OCaml is open source. The main compiler/runtime implementation is public and actively maintained. 

Yes, there are open-source implementations/variants, but the main upstream implementation is already open, so that is the one to study first. 

Building a full OCaml-compatible compiler from scratch is hard.

Building a small OCaml-like language is very feasible.

For a fresh implementation, use Rust or C.

OCaml is bootstrapped/self-hosting, but it still relies on a C runtime and bootstrap artifacts, so it is not quite the same style as a tiny classic Lisp or Forth bootstrap.

license(s)?

Short answer
The main OCaml implementation is open source under the LGPL v2.1 (with a special linking exception).

Parts of the toolchain and ecosystem use more permissive licenses (BSD / MIT / ISC).

The compiler runtime and standard library are explicitly designed to be usable in proprietary programs without forcing your code to be open source.

1. Core OCaml license (what matters most)
LGPL v2.1 + OCaml linking exception
The OCaml compiler, runtime, and standard library are licensed under:

GNU LGPL v2.1

PLUS a special exception

What the exception does
Normally LGPL has requirements around linking (especially static linking).
OCaml adds an explicit exception that says (in plain terms):

You can link OCaml runtime / stdlib into your program without having to open-source your program, even if statically linked.

Practical implications
Scenario	Allowed?	Notes
Closed-source app using OCaml	✅	Fully allowed
Static linking OCaml runtime	✅	Exception explicitly allows this
Modifying OCaml compiler/runtime	⚠️	You must publish those changes
Using OCaml libraries	✅	Usually permissive
This makes OCaml much closer in practice to MIT/BSD-style usability than raw LGPL.

2. Other parts of the ecosystem
Different components use different licenses:

Compiler tools & runtime
Mostly LGPL v2.1 + exception

Standard library
Same as above (LGPL + exception)

Auxiliary tools (varies)
Some are:

BSD

MIT

ISC

Package ecosystem (OPAM)
Mixed licenses (depends on package author)

Many libraries are MIT/BSD

3. Comparison to other languages
Language	License model	Practical restriction level
C/C++ (gcc/clang)	GPL/LGPL mix	Medium
Rust	MIT/Apache-2.0	Very permissive
OCaml	LGPL + exception	Effectively permissive
Java (OpenJDK)	GPL + classpath exception	Similar idea
Lisp (varies)	Often MIT/BSD	Very permissive
OCaml’s model is closest to:

“copyleft for the compiler itself, permissive for users of the language.”

4. If you build your own OCaml implementation
You have choices:

Clean-room implementation
You can use:

MIT

BSD

Apache-2.0

GPL/LGPL if you want copyleft

No restriction from OCaml itself if you don’t copy code.

Reusing OCaml code
You must comply with:

LGPL v2.1

linking exception terms

Modifications must be shared

5. Key architectural takeaway
From a systems-design perspective:

OCaml deliberately separates:

language usability (permissive)

compiler/runtime modifications (copyleft)

This is similar to:

Java’s Classpath exception

GCC runtime exception

It enables:

commercial adoption

static binaries

embedded use

without license friction

Bottom line
OCaml is open source and business-friendly

License = LGPL v2.1 + special exception

You can:

build proprietary software with it

statically link it

You only need to open-source:

changes to the compiler/runtime itself

Does OCaml support FFI?
Yes
Mainline OCaml explicitly supports a foreign-function interface via external declarations and user-defined primitives implemented in lower-level code. The official manual’s “Interfacing C with OCaml” chapter describes exactly that: C functions can be linked with OCaml code and called from OCaml functions, and C can also call back into OCaml. 

In stock OCaml, it looks like this conceptually:

external uart_putc : int -> unit = "caml_uart_putc"
external uart_getc : unit -> int = "caml_uart_getc"
On a normal host build, those names map to C runtime functions. 

For your subset

VM opcodes

interrupt-driven ring buffer services

So in your OCaml-on-COR24, “FFI” probably really means:

binding OCaml primitives to runtime entry points implemented in your Pascal+p-code+VM runtime

not binding to hosted C libraries.

App-facing level

getc pops from RX queue

putc queues to TX queue or blocks if full

Interpreted mode vs compiled mode: how I/O differs
This is important in your architecture.

In interpreted mode
The OCaml interpreter itself recognizes primitive calls like Board.switch () and dispatches them to runtime functions.

Runtime path:

COR24 native
  -> p-code VM
    -> Pascal OCaml interpreter
      -> primitive dispatch
        -> board/UART runtime
          -> MMIO / buffers / interrupts
So switch () is interpreted as a primitive call, not as ordinary user code.

In compiled mode
Your OCaml compiler emits either:

p-code calls to runtime entry points

or bytecode ops that the runtime maps to board/UART services

Runtime path:

COR24 native
  -> p-code VM
    -> compiled OCaml p-code
      -> runtime helper / primitive
        -> MMIO / buffers / interrupts
So in compiled mode, I/O calls become more direct. No AST-walking interpreter is involved.

Difference
Interpreted mode: primitive lookup/dispatch happens through the OCaml interpreter

Compiled mode: primitive call is already lowered into a direct runtime call

That is one reason compiled mode will feel much cleaner for embedded I/O.
